Abstract

One might have noticed that Belgian politicians are currently in “election mode”. End of May 2019, the masses will set direction towards an Electoral Office to cast their vote. People need to volunteer to supervise the correct voting procedures, and all votes are manually counted. All of these procedures currently deployed might seem like a task screaming for automation and benefiting from a web-based approach.

However, it is key to thoroughly analyze the viability of such a system first. Your task is to investigate what it would take, if at all possible, to implement a web-based electronic voting system in a secure, fair and democratic manner. The goal of this assignment is to focus on the security aspects of online elections, and the arguments against or in favour of such an approach in comparison to the traditional voting process.

1 Introduction

Login using eID, assuming that the country has an eID system.

A Belgian citizen, should use his identity card to log in on to the website. Once logged-in, he could make his vote and save it. The website will keep a list of all votes, while keeping it anonymous and it will also keep a list of the people that did vote.

The following assumptions are made:

  • A voter can only cast their vote once.
  • Every voter has access to an eID reader or is welcome to use a voting system provided by the government.
  • No proxy voting can be performed.
  • The network will be available during the voting period without interruption.
  • The power supply won’t be interrupted.

The proof-of-concept demonstration website is created using Node.js and the Express package. For the database MongoDB is used with the mongodb package (https://mongodb.github.io/node-mongodb-native/). To ensure security and encryption the following packages are used:

  • crypto (built-in) (https://nodejs.org/api/crypto.html)
  • https (https://nodejs.org/api/https.html)
  • node-rsa (https://www.npmjs.com/package/node-rsa)

2 Services

1.
Confidentiality
(a)
The vote should be private.
(b)
Backtracking to the author should not be possible.
2.
Authentication
(a)
Only one vote per person can be cast.
(b)
Someone should not be be able to vote on one’s behalf.
3.
Access control / authorization
(a)
The user must be entitled to vote.
(b)
Voting should happen within allowed time span.
4.
Data integrity
(a)
The table is sticky, a cast vote cannot be changed.
(b)
One’s vote should not be counted twice.
5.
Non-repudiation
(a)
Whether or not a person has voted, not which party was voted for.
(b)
From government’s perspective a vote can not be considered invalid if it has been correctly cast.
(c)
The user should have a certificate to prove he has voted (in case the server database crashes)...
i.
The server makes a certificate with the users credentials.
ii.
The server signs this certificate using his private key.
iii.
The server encrypts the certificate using the users public key.
iv.
The user receives the certificate for later proof.
6.
Availability
(a)
Must be accessible during the complete voting day, since cast votes can’t be lost.
(b)
The websites/databases will be decentralized. Multiple servers will set-up, in case a server is down the other can be accessed and the voting can continue.
(c)
Some public spaces should be provided with computers for people that do not have direct access to an internet connection.

3 Attacks

  • Passive attacks
    1.
    Eavesdropping
    (a)
    A third party wants to leak the current intermediate election results to influence the further voting process.
    2.
    Traffic analysis
    (a)
    Not applicable.
  • Active attacks
    1.
    Message insertion/modification:
    (a)
    A third party want to change votes to a certain political party.
    2.
    Impersonation/masquerade:
    (a)
    A third party want to repeat votes on a certain political party.
    (b)
    A third party want to reduce the votes for certain political parties.
    3.
    Replay:
    (a)
    A third party want to repeat votes on a certain political party.
    4.
    (Distributed) Denial-of-Service ((D)Dos):
    (a)
    A third party wants to boycott the elections.
    5.
    Hijacking (taking over existing connection):
    (a)
    Man-in-the-middle (MITM) attacks: a third party pretends to be the voting website to vote with the visitors credentials.
    (b)
    Man-in-the-browser (MITB) attacks: malware that attacks the website in the user’s browser.

4 Countermeasures

  • Passive attacks
    1.
    Eavesdropping
    (a)
    Encryption of authentication data and the user’s vote.
    2.
    Traffic analysis
    (a)
    Not applicable.
  • Active attacks
    1.
    Message insertion/modification:
    (a)
    Digital signature of the user.
    2.
    Impersonation/masquerade:
    (a)
    Digital signature with private key from eID.
    3.
    Replay:
    (a)
    Server-side package dropping based on presence of entry in database.
    4.
    (Distributed) Denial-of-Service ((D)Dos):
    (a)
    Use third party service that prevents this issue to redirect traffic to the voting application.
    5.
    Hijacking (taking over existing connection):
    (a)
    SSL/TLS certificates used in the HTTPS protocol.

5 Limitations and Vulnerabilities

  • Tables in database with votes can easily be altered by trusted people with authorized access. This is no different from the current voting system and this problem has not been solved by this implementation.
  • Someone can still steal a user’s eID and threaten them in order to get their PIN code.
  • The system, as any web application, can only work if a stable power source and internet connection can be provided to the user. This responsibility does not lie with us, but with the power and telecom providers in Belgium.
